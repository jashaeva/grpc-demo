sh localenv.sh
~~~

### Запустить сервисы
Настроить конфигурацию запуска Spring Boot сервиса (Application) и запустить
- grpc-server (сам сервер)
- grpc-client (этот только тестирует)

**При изменении контракта сервиса**
Надо пересобрать проект grpc-common (grpc-common -> Tasks ->build -> build)

### Важные замечания для тестировщика
**При тестировании gRPC сервисов** учитывайте следующие нюансы:
1. **Стандарт vs Реальность**: документация разделяет коды на "генерируемые библиотекой" (например, UNAVAILABLE при потере соединения) и "возвращаемые бизнес-логикой" (например, NOT_FOUND).
Тестировщику важно проверять оба сценария. Проверка в коде: в отличие от HTTP, где код в строке браузера, gRPC код проверяется в коде клиента. Например, на JavaScript это выглядит так:
~~~ javascript
pm.test('Status code is 0', () => {
   pm.response.to.have.statusCode(0);
});
~~~
1. **Метаданные**: помимо кода, всегда проверяйте **Trailers**. Критическая информация об ошибке (например, детали превышения квоты) часто передается именно там, а не в теле ответа.
 

## Метаданные (Headers)
**Что это**: невидимые для тела запроса/ответа пары "ключ-значение", которые передаются до начала основного сообщения.
Когда отправляются:
 * Клиент → Сервер: Перед отправкой запроса (в начале стрима)
 * Сервер → Клиент: В начале ответа (до первого сообщения)

**Примеры использования**:
 * authorization: Bearer token123
 * user-agent: grpc-python/1.45
 * request-id: abc-123-def
 * x-api-version: v2

**Аналог в HTTP**: обычные HTTP-заголовки (но в HTTP/2 они бинарные и сжимаются).

## Trailers (Трейлеры)
**Что это**: заголовки, которые отправляются после всех сообщений, прямо перед закрытием стрима.

**Особенность**: в gRPC код статуса и сообщение об ошибке всегда передаются в трейлерах, даже при успешных запросах.

**Аналог в HTTP**: нет прямого аналога в HTTP/1.1 (трейлеры появились в HTTP/2). В обычном REST API статус-код идет в первой строке ответа.

## Почему это важно для тестировщика
| Что вы видите в Postman / gRPC клиенте     | Где это на самом деле |
|--------------------------------------------|-----------------------|
| grpc-status: 5                             | В трейлерах           |
| grpc-message: "User not found"             | В трейлерах           |
| grpc-status-details-bin (бинарные детали)  | В трейлерах           |

Пример трейлеров в сыром виде (через Wireshark/логи):
~~~text
trailer: grpc-status: 5
trailer: grpc-message: User with id 123 not found
trailer: custom-error-code: E12345
~~~

### Что проверять тестировщику
1. Не доверяйте только телу ответа — при ошибке тело может быть пустым, а вся диагностика в трейлерах.
1. Проверяйте обязательные трейлеры (всегда должны быть): 
   * grpc-status (код 0-16)
   * grpc-message (текст ошибки, если статус ≠ 0)
1. Дополнительные трейлеры (зависят от API):
   * grpc-status-details-bin (protobuf-сериализованные детали ошибки)
   * Кастомные ключи для трассировки/метрик
1. В Postman / Insomnia: обычно трейлеры автоматически маппятся в специальные переменные ответа. Но для отладки включайте "Raw view" или смотрите консоль.

### Распространенные грабли
| Проблема                            | Что происходит                                                                                                               |
|-------------------------------------|------------------------------------------------------------------------------------------------------------------------------|
| Стриминг с ошибкой посреди передачи | Первые сообщения дошли, потом ошибка пришла в трейлерах (но вы её не увидите, пока стрим не закроется)                       |
| gRPC-web (браузер)                  | Браузеры не умеют читать трейлеры напрямую, поэтому прокси-слои (Envoy, gRPC-web proxy) перекладывают их в обычные заголовки |
| Размер трейлеров                    | Технически лимит 16KB на ключ+значение (как и у хедеров HTTP/2)                                                              |


### Практический пример проверки в коде (JS)
~~~javascript
// В тестах Postman (после получения ответа)
pm.test("Status code is OK", () => {
   pm.response.to.have.statusCode(0); // 0 = OK
});

pm.test("Trailers contain custom error code", () => {
// В некоторых клиентах трейлеры доступны через pm.response.headers
   const errorCode = pm.response.headers.get("custom-error-code");
   pm.expect(errorCode).to.equal("E12345");
});
~~~

**Коротко**: метаданные = заголовки в начале, трейлеры = заголовки в конце (там где живет gRPC-статус). Тестировщик должен проверять оба типа заголовков, особенно при ошибках.
